# 회의록 자동화 서비스 — 개요 설계 (Overview)

- 작성일: 2026-07-10
- 상태: 초안 승인 완료 (개요 수준). 단계별 상세 스펙은 후속 문서로 분리 예정.
- 관련 문서: 없음 (최초 문서)

## 1. 목적 & 배경

팀원들이 별도 앱 설치나 유료 노션 구독 없이, 웹에서 회의를 녹음하고 AI 요약본을 확인·보관할 수 있는 사내 전용 생산성 툴을 구축한다. 핵심 가치는 무료 티어 조합으로 유료 서비스 수준의 팀 전용 AI 대시보드/아카이빙 시스템을 확보하는 것이다.

## 2. 성공 기준

- 팀원이 회의 후 직접 요약본을 작성하지 않아도 된다.
- 과거 회의 내용을 검색/챗봇으로 빠르게 찾을 수 있다.
- 노션 없이도 이 대시보드만으로 팀이 회의록을 관리한다.
- 팀원들이 실제로 꾸준히 사용한다 (채택률이 최종 지표).
- 필요한 사람은 노션·슬랙으로 내보낼 수 있다 (옵션, 강제 아님).

## 3. 사용 규모 및 제약

- 현재 팀 규모/회의 빈도는 미정이며, 전사 확장 가능성이 있다. 처음부터 특정 규모에 못박히지 않는 구조로 설계한다.
- 보안 요건: 회의 원본 음성/스크립트가 제3자 SaaS에 영구 저장되는 것은 지양한다. 단, AI 요약(Gemini API 호출) 자체는 API 특성상 별도 학습 재사용이 되지 않는 경우가 일반적이라 문제로 보지 않는다. 즉 "데이터를 어디에 두느냐"가 핵심 제약이지 "AI를 쓰느냐"는 아니다.
- 용량 문제 및 보안을 동시에 해결하기 위해, **원본 음성은 STT 처리 직후 폐기**하고 텍스트(스크립트·요약·액션아이템)만 영구 보관한다.
  - 트레이드오프: STT 오인식을 나중에 원음으로 재검증할 수 없다. 성공 기준에 재청취 요구가 없으므로 수용 가능한 트레이드오프로 판단.

## 4. 녹음 및 공유 모델

- 회의당 대표 1인이 브라우저로 녹음 (오디오 파일 1개, 화자 분리 없음).
- 녹음자가 참석자를 태깅한다.
- 회의록은 녹음자만이 아니라 **태깅된 참석자 전원**의 대시보드에 노출된다 (녹음자 = 참석자 중 1인일 뿐).
- 다중 참여자 각자 개별 녹음 후 병합하는 방식(화상회의 참여자별 동시 녹음)은 범위 밖.

## 5. 아키텍처 & 데이터 흐름

노코드 우선, RAG/챗봇처럼 실제 로직이 필요한 지점만 코드(바이브코딩)로 구현하는 하이브리드 구조.

```
[1. Softr] 로그인 → 녹음 버튼 → 참석자 태깅
      ↓
[2. Make 웹훅] 오디오 blob 직접 전송 (중간 저장소 없음)
      ↓
[3. Gemini API] STT 스크립트 + 3줄 요약 + 액션아이템 생성
      ↓
[4. Make] 처리 완료 후 오디오 즉시 폐기 (Make 시나리오 데이터 로그 보관 최소화 설정 포함)
      ↓
[5. Xano] 텍스트만 영구 저장 (스크립트·요약·액션아이템·참석자ID) — 무료 플랜 기준 레코드 10만 건까지 지원
      ↓
[6. Softr 대시보드] Xano를 데이터소스로 연결, 참석자 태깅된 회의만 유저별로 필터링해 노출
      ↓
[7. RAG 챗봇 위젯] (바이브코딩, Softr에 임베드) Xano API로 텍스트를 가져와 인덱싱, 자연어 질의응답
      ↓
[8. 노션/슬랙 내보내기] Softr 버튼 → Make → Notion API / Slack API (옵션, 유저가 원할 때만)
```

### 컴포넌트별 역할

| 컴포넌트 | 역할 | 구현 방식 |
|---|---|---|
| Softr | 인증, 녹음 UI, 대시보드, 상세페이지, 내보내기 버튼 | 노코드 |
| Make | 이벤트 기반 오케스트레이션 (수신→AI호출→DB기록→내보내기 트리거) | 노코드 |
| Gemini API | 오디오 STT + 요약 + 액션아이템 추출 | API 호출 |
| Xano | 텍스트 데이터의 영구 저장소 (DB) | 노코드 백엔드 |
| RAG 챗봇 위젯 | 과거 회의록 자연어 질의응답 | 코드 (바이브코딩) — 유일한 코드 영역 |

### 주요 기술 결정과 근거

- **DB: Xano (Airtable 아님)** — Airtable 무료 플랜은 베이스당 1,000레코드 하드캡이라 전사 확장 시나리오에서 조기에 막힐 위험이 있음. Xano는 레코드 개수 캡이 없고 무료 플랜 기준 10만 레코드까지 지원(용량/API 호출량 기준 과금)해 이 서비스(텍스트 위주)에서는 사실상 제약이 되지 않음. Softr가 Xano를 네이티브 데이터소스로 지원하므로 노코드 전제도 유지됨.
- **원본 오디오 미저장** — Google Drive 등 중간 저장 단계 없이 Make 웹훅으로 오디오를 직접 수신해 Gemini에 전달하고, 처리 후 폐기. Make의 시나리오 실행 로그 보관 기간이 기본 설정상 일정 기간 남을 수 있으므로, 로그 보관을 최소화하는 설정을 파이프라인 구축 시 함께 적용한다.
- **챗봇만 코드 영역** — Softr/Make/Xano 조합으로는 RAG(임베딩+벡터검색) 기능을 구현할 수 없어, 이 부분만 별도 경량 백엔드(바이브코딩)로 구축한다.

## 6. 로드맵

1. **Xano DB 설계**: 회의록 테이블(제목/스크립트/요약/액션아이템/참석자 관계) + 유저 테이블 설계.
2. **AI 파이프라인 검증**: Make 웹훅 → Gemini STT+요약 → Xano 저장, 오디오 폐기까지 엔드투엔드 테스트. Make 로그 보관 최소화 설정 포함.
3. **Softr UI + 권한**: 회원가입/로그인, 녹음 버튼, 참석자 태깅 UI, "내가 참석자인 회의" 필터링 대시보드. Xano를 데이터소스로 연동.
4. **RAG 챗봇** (바이브코딩): 기술 스택 및 임베딩/인덱싱 방식은 별도 상세 스펙에서 결정.
5. **내보내기**: 노션/슬랙 버튼 → Make → 각 API 연동.

## 7. 범위 밖 (Out of Scope)

- 화자 분리(다중 참여자 개별 녹음 및 병합)
- 오디오 원본 재청취 기능
- 이미지/파일 첨부 등 텍스트 외 자료 저장

## 8. 검증이 필요한 리스크

- Softr↔Xano 실제 연동 품질 및 데이터소스 필터링 동작은 3단계에서 실증 필요.
- 챗봇 위젯을 Softr 페이지에 임베드하는 구체적 방식은 4단계에서 결정 필요.
- Make 무료 티어(월 1,000 오퍼레이션)는 전사 확장 시 유료 전환 검토 대상.
- Xano/Airtable/Make 등 각 툴의 무료 플랜 세부 조건(용량, 호출량 한도)은 시점에 따라 변경될 수 있어, 실제 구축 단계에서 최신 요금제를 재확인해야 한다.
