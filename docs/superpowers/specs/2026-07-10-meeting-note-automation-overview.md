# 회의록 자동화 서비스 — 개요 설계 (Overview)

- 작성일: 2026-07-10
- 상태: 초안 승인 완료 (개요 수준). 단계별 상세 스펙은 후속 문서로 분리 예정.
- 관련 문서: 없음 (최초 문서)

## 1. 목적 & 배경

팀원들이 별도 앱 설치나 유료 노션 구독 없이, 웹에서 회의를 녹음하고 AI 요약본을 확인·보관할 수 있는 사내 전용 생산성 툴을 구축한다. 핵심 가치는 무료 티어 조합으로 유료 서비스 수준의 팀 전용 AI 대시보드/아카이빙 시스템을 확보하는 것이다.

## 2. 성공 기준

- 팀원이 회의 후 직접 요약본을 작성하지 않아도 된다.
- 과거 회의 내용을 검색/챗봇으로 빠르게 찾을 수 있다.
- 노션 없이도 이 대시보드만으로 팀이 회의록을 관리한다.
- 팀원들이 실제로 꾸준히 사용한다 (채택률이 최종 지표).
- 필요한 사람은 노션·슬랙으로 내보낼 수 있다 (옵션, 강제 아님).

## 3. 사용 규모 및 제약

- 현재 팀 규모/회의 빈도는 미정이며, 전사 확장 가능성이 있다. 처음부터 특정 규모에 못박히지 않는 구조로 설계한다.
- 보안 요건: 회의 원본 음성/스크립트가 제3자 SaaS에 영구 저장되는 것은 지양한다. 단, AI 요약(Gemini API 호출) 자체는 API 특성상 별도 학습 재사용이 되지 않는 경우가 일반적이라 문제로 보지 않는다. 즉 "데이터를 어디에 두느냐"가 핵심 제약이지 "AI를 쓰느냐"는 아니다.
- 용량 문제 및 보안을 동시에 해결하기 위해, **원본 음성은 STT 처리 직후 폐기**하고 텍스트(스크립트·요약·액션아이템)만 영구 보관한다.
  - 트레이드오프: STT 오인식을 나중에 원음으로 재검증할 수 없다. 성공 기준에 재청취 요구가 없으므로 수용 가능한 트레이드오프로 판단.

## 4. 녹음 및 공유 모델

- 회의당 대표 1인이 브라우저로 녹음 (오디오 파일 1개, 화자 분리 없음).
- 녹음자가 참석자를 태깅한다.
- 회의록은 녹음자만이 아니라 **태깅된 참석자 전원**의 대시보드에 노출된다 (녹음자 = 참석자 중 1인일 뿐).
- 다중 참여자 각자 개별 녹음 후 병합하는 방식(화상회의 참여자별 동시 녹음)은 범위 밖.

## 5. 아키텍처 & 데이터 흐름

노코드 우선, 로컬 STT 처리와 RAG/챗봇처럼 실제 로직이 필요한 지점만 코드(바이브코딩)로 구현하는 하이브리드 구조.

**2026-07-10 개정**: 기성 회의록 AI 툴(Otter/Fireflies/Fathom/Zoom AI Companion 등) 비교 리서치 결과, 오프라인 대면 회의 녹음 용도로 딱 맞는 툴이 없었고(대부분 화상회의 봇 참여 방식), 이미 라이선스·로컬 실행을 확인한 오픈소스 Whisper 파인튜닝 모델(`batiai/stt`)을 자체 서버에서 돌리는 쪽으로 STT 구현 방식을 변경. 이에 따라 Make+Gemini가 맡던 STT 역할이 자체 처리 서버로 이동.

```
[1. Softr] 로그인 → 녹음 버튼 → 참석자 태깅
      ↓
[2. 자체 처리 서버] (바이브코딩, 로컬/자체 호스팅) 오디오 직접 수신
      ↓
[3. batiai/stt (로컬 Whisper 파인튜닝)] 오디오 → 스크립트 전사, 처리 후 오디오 즉시 폐기
      ↓
[4. Gemini API] 스크립트 텍스트 → 3줄 요약 + 액션아이템 생성 (텍스트만 전송, 음성 미전송)
      ↓
[5. Xano] 텍스트만 영구 저장 (스크립트·요약·액션아이템·참석자ID) — 무료 플랜 기준 레코드 10만 건까지 지원
      ↓
[6. Softr 대시보드] Xano를 데이터소스로 연결, 참석자 태깅된 회의만 유저별로 필터링해 노출
      ↓
[7. RAG 챗봇 위젯] (바이브코딩, Softr에 임베드) Xano API로 텍스트를 가져와 인덱싱, 자연어 질의응답
      ↓
[8. 노션/슬랙 내보내기] Softr 버튼 → Make → Notion API / Slack API (옵션, 유저가 원할 때만)
```

### 컴포넌트별 역할

| 컴포넌트 | 역할 | 구현 방식 |
|---|---|---|
| Softr | 인증, 녹음 UI, 대시보드, 상세페이지, 내보내기 버튼 | 노코드 |
| 자체 처리 서버 | 오디오 수신 → 로컬 STT 실행 → 오디오 폐기 → Gemini 요약 호출 → Xano 저장까지 파이프라인 전체 소유 | 코드 (바이브코딩) |
| batiai/stt | 로컬 Whisper 파인튜닝 STT 모델 (상업용 라이선스·로컬 실행 확인됨) | 오픈소스, 자체 처리 서버에서 실행 |
| Gemini API | 스크립트 텍스트 → 요약 + 액션아이템 추출 (텍스트만, 음성 없음) | API 호출 |
| Xano | 텍스트 데이터의 영구 저장소 (DB) | 노코드 백엔드 |
| RAG 챗봇 위젯 | 과거 회의록 자연어 질의응답 | 코드 (바이브코딩) |
| Make | 노션/슬랙 내보내기 트리거만 담당 | 노코드 |

### 주요 기술 결정과 근거

- **DB: Xano (Airtable 아님)** — Airtable 무료 플랜은 베이스당 1,000레코드 하드캡이라 전사 확장 시나리오에서 조기에 막힐 위험이 있음. Xano는 레코드 개수 캡이 없고 무료 플랜 기준 10만 레코드까지 지원(용량/API 호출량 기준 과금)해 이 서비스(텍스트 위주)에서는 사실상 제약이 되지 않음. Softr가 Xano를 네이티브 데이터소스로 지원하므로 노코드 전제도 유지됨.
- **STT: 로컬 오픈소스 모델(batiai/stt)** — 기성 SaaS 회의록 툴은 대부분 화상회의 봇 참여 방식이라 오프라인 대면 회의 녹음 용도에 안 맞고, 데이터 거버넌스 리스크(예: Otter/Fireflies의 진행 중인 집단소송)도 있어 자체 처리 서버에서 로컬 STT를 실행하는 쪽으로 결정. 오디오가 처음부터 끝까지 외부로 전송되지 않아 원본 미저장 원칙이 훨씬 단순하게 달성됨(제3자 로그 잔존 리스크 자체가 없음).
- **요약: Gemini API 유지** — STT로 이미 텍스트화된 스크립트만 전송하므로 음성 정보(화자 목소리 등) 유출 리스크가 없고, 요약/액션아이템 품질이 로컬 오픈소스 LLM보다 안정적이라 판단.
- **코드 영역 확장** — 원래는 챗봇만 코드로 구현할 계획이었으나, STT를 로컬로 옮기면서 자체 처리 서버(파이프라인 오케스트레이션 포함) 전체가 코드 영역이 됨. Softr(UI)·Xano(DB)·Make(내보내기)는 여전히 노코드로 유지.

## 6. 로드맵

1. **Xano DB 설계**: 회의록 테이블(제목/스크립트/요약/액션아이템/참석자 관계) + 유저 테이블 설계.
2. **AI 파이프라인 검증**: 자체 처리 서버 골격 구축(오디오 수신 → batiai/stt 로컬 전사 → 오디오 폐기 → Gemini 요약 호출 → Xano 저장)까지 엔드투엔드 테스트.
3. **Softr UI + 권한**: 회원가입/로그인, 녹음 버튼, 참석자 태깅 UI, "내가 참석자인 회의" 필터링 대시보드. Xano를 데이터소스로 연동.
4. **RAG 챗봇** (바이브코딩): 기술 스택 및 임베딩/인덱싱 방식은 별도 상세 스펙에서 결정.
5. **내보내기**: 노션/슬랙 버튼 → Make → 각 API 연동.

## 7. 범위 밖 (Out of Scope)

- 화자 분리(다중 참여자 개별 녹음 및 병합)
- 오디오 원본 재청취 기능
- 이미지/파일 첨부 등 텍스트 외 자료 저장

## 8. 검증이 필요한 리스크

- ~~브라우저 마이크 녹음(getUserMedia+MediaRecorder)이 기술적으로 되는지~~ — **2026-07-10 스파이크로 검증 완료**. `spikes/recording-spike.html`에서 녹음→재생→다운로드까지 확인됨(웹킷 계열은 MediaRecorder blob에 duration 메타데이터가 없어 재생 UI가 0:00으로 보이는 알려진 이슈가 있었으나 강제 계산 트릭으로 해결). 다음 확인 대상: 이 방식을 Softr에 실제로 임베드할 수 있는지, 모바일 브라우저(HTTPS 필요)에서도 동일하게 되는지.
- Softr↔Xano 실제 연동 품질 및 데이터소스 필터링 동작은 3단계에서 실증 필요.
- 챗봇 위젯을 Softr 페이지에 임베드하는 구체적 방식은 4단계에서 결정 필요.
- 자체 처리 서버의 운영 부담(어디서 상시 구동할지 — 로컬 상시 구동 PC/Mac vs 클라우드 VM, 회의 시각에 서버가 꺼져 있으면 파이프라인 자체가 동작 안 함)은 2단계에서 확정 필요.
- batiai/stt의 실제 한국어 회의 전사 정확도(다자 발화·잡음 환경)는 2단계 파이프라인 검증에서 실측 필요.
- Make는 이제 내보내기 기능에만 쓰이므로 무료 티어 오퍼레이션 한도 문제는 사실상 해소됨.
- Xano/Airtable/Make 등 각 툴의 무료 플랜 세부 조건(용량, 호출량 한도)은 시점에 따라 변경될 수 있어, 실제 구축 단계에서 최신 요금제를 재확인해야 한다.

## 9. 기성 툴 검토 결과 (참고)

자체 구축 결정 전에 Otter.ai, Fireflies.ai, Fathom, tl;dv, Zoom AI Companion 등 기성 AI 회의록 툴을 리서치로 비교했음. 요약:

- **Otter.ai**: 한국어 STT 미지원, 고객 음성/전사본을 자체 AI 학습에 기본 사용(옵트아웃 필요), 비동의 녹음·음성지문 수집 관련 연방 집단소송 4건 진행 중(2026-07 기준) — 제외.
- **Fireflies.ai**: 한국어 지원 양호, AskFred 챗봇으로 과거 회의 자연어 질의 가능, Slack 연동 전 플랜 지원 — 요구사항은 가장 폭넓게 충족하나 음성지문 보관정책 미공개로 일리노이 BIPA 소송 2건 진행 중.
- **Fathom**: 무료 플랜에서 녹음·전사·저장 영구 무제한 — 단 한국어 요약은 미지원(전사만 한국어, 요약은 영어 등 6개 언어로만 제공).
- **Zoom AI Companion**: 고객 콘텐츠를 AI 학습에 쓰지 않는다는 공식 정책으로 거버넌스가 가장 안전하나, 다른 조건(가격/한국어) 미검증.
- 조사한 툴 대부분이 화상회의(Zoom/Meet/Teams) 봇 참여 방식이 기본 동작이라, 팀의 주된 회의 형태인 **오프라인 대면 회의 녹음**에는 적합하지 않음이 확인됨. 이 점이 자체 구축으로 최종 결정하게 된 핵심 근거.
